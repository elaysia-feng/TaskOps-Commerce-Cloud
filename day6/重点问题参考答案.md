# 重点问题参考答案（P0）

## 1. 3 分钟讲项目与职责
我负责 TaskOps Cloud 后端主流程与跨服务治理，覆盖用户侧 `注册/登录 -> 创建任务 -> 条件检索 -> 热门任务`，以及管理侧聚合看板。技术上我主导了 `Gateway 统一 JWT 鉴权 + 用户上下文透传`、`task-service 的 MyBatis-Plus 分页动态查询`、`Redis ZSet 热榜与缓存防护`、`admin-service 通过 OpenFeign 聚合 auth/task/order/pay`，并落地 `Nacos` 的注册发现与配置管理。

## 2. 为什么统一鉴权放 Gateway
把鉴权前置到网关可以避免每个服务重复解析 JWT，减少重复代码和策略漂移。所有请求先过一层统一规则（白名单、401/403、角色校验），安全口径一致，下游服务更聚焦业务。

## 3. 白名单如何做，误配后果
用路径匹配（如 `/api/auth/**`、`/api/tasks/hot`）直接放行，其余接口必须带 Bearer Token。误把敏感接口放进白名单会导致未登录可访问，属于高危配置错误；我会配合自动化用例校验“白名单最小化”。

## 4. 为什么透传 X-User-* 头
网关已经完成 token 解析，下游直接用 `X-User-Id/X-Username/X-Roles` 即可，避免重复验签。这样延迟更低，也统一了用户上下文读取方式。

## 5. 为什么 ADMIN 二次校验
网关做第一道拦截，admin-service 再做业务兜底，属于纵深防御。即使网关规则被误改或被绕过，下游仍会拒绝非 ADMIN 请求。

## 6. 直连下游伪造头怎么防
生产上应只允许内网或服务网关访问下游服务（网络层封禁外部直连）。同时下游对内部入口做来源校验（如网关签名头或 mTLS），避免信任任意 `X-User-*`。

## 7. JWT 是否支持主动下线
纯 JWT 默认不支持“立刻失效”，要靠过期时间。要支持踢人可加 `jti + Redis 黑名单`，或维护 token version（用户级版本号）做二次校验。

## 8. 登录失败锁定怎么实现
Redis 记录失败次数键 `auth:fail:{username}`（10 分钟过期），达到阈值后写锁键 `auth:lock:{username}`（5 分钟）。登录前先检查锁键，命中直接拒绝。

## 9. 阈值为什么是 5 次/10 分钟、锁 5 分钟
这是安全与可用性的折中：能有效抬高暴力破解成本，又不至于误伤普通用户太久。生产会按风控数据动态调整，而不是写死不变。

## 10. 创建任务为什么先调 auth 校验 owner
防止脏数据和伪造 ownerId。即使前端传参异常，服务端也能通过远程校验保证“任务归属用户真实存在”。

## 11. 为什么用 MyBatis-Plus 动态查询
查询维度多（关键词/状态/优先级区间），用 LambdaWrapper 可读性高、类型安全、扩展快，避免手写字符串拼 SQL 易错。

## 12. like 检索的瓶颈
数据量大时 `%keyword%` 难用索引，会导致扫描放大和响应抖动。后续可按规模升级到全文索引或 ES。

## 13. 分页边界保护
页码最小 1，size 设上限（如 50），防止恶意大页拖垮数据库和 JVM 内存。属于基础限流手段之一。

## 14. 详情“读后写”设计利弊
优点是能自然积累活跃度信号（进度/更新时间）用于热榜。缺点是读请求放大成写，热点任务会增加 DB 压力；线上可改为异步计数或批量回写。

## 15. 热榜为什么用 Redis ZSet
ZSet 天然支持按 score 排序和 TopN 查询，复杂度低且性能稳定。比 List 更适合动态排序，比 Hash 更适合取排名。

## 16. 热榜分数如何解释
当前按 `质量分 + 进度权重 + 评论权重 + 优先级修正` 计算，目标是平衡“内容质量”和“近期活跃度”。权重来自业务经验，后续可基于点击/转化数据迭代。

## 17. 空榜触发重建的并发问题
现在存在并发重建风险（多个请求同时发现空榜）。优化可加分布式锁或单飞（single-flight）机制，保证同一时刻只有一个重建任务。

## 18. 穿透/击穿/雪崩如何处理
穿透：缓存空值（`__NULL__` + 短 TTL）。
击穿：互斥锁 `setIfAbsent + 过期时间`，只放一个线程回源。
雪崩：TTL 随机抖动，打散集中失效。

## 19. 互斥锁方案风险点
锁过期与业务耗时不匹配会导致并发回源；递归重试在极端高并发下会放大等待。进阶可用 Lua 释放锁、续约机制、限次重试。

## 20. 为什么线程池拒绝策略用 CallerRunsPolicy
队列满时让调用线程回退执行，形成反压，避免任务直接丢失。代价是入口请求延迟会升高，但系统整体更稳。

## 21. 看板聚合超时时整体失败还是部分可用
管理看板更适合“部分可用”，即主结构返回、失败项标记降级信息。这样可用性高于强一致全失败，便于运营决策。

## 22. 目前聚合缺什么，怎么补
当前主要是直连聚合，需补 Feign 超时、重试、熔断与 fallback，外加统一超时预算和降级响应结构。

## 23. Nacos 具体作用
一是服务注册发现（`lb://service-name` 路由与调用）；二是配置中心（公共配置、环境差异配置集中管理）。

## 24. 配置中心不可用如何兜底
启动时可用本地默认值和 `optional:nacos` 兜底，运行期要有本地缓存配置与告警。关键参数（如 JWT 密钥）必须有安全回退策略。

  1. 配置兜底

  - 关键配置本地默认值 + 环境变量兜底，不把生死配置只放 Nacos。

  2. 高可用部署

  - Nacos 集群 + MySQL 持久化 + 负载均衡，不用单机。

  3. 客户端容错

  - 服务调用加超时、重试、熔断、降级，避免把“Nacos抖动”放大成全链路雪崩。

  4. 运维预案

  - 监控 Nacos 可用性、注册实例数、配置推送失败率；异常时切只读/冻结变更，优先保运行稳定。

## 25. 如何证明抗压能力
用压测和监控说话：对比改造前后 `QPS/P95/DB CPU/Redis 命中率`。目标是让高频读优先走 Redis，数据库只承担必要回源与写路径。
