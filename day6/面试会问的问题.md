# 面试会问的问题（TaskOps Cloud）

## 一、重点必问（P0）
1. 你先用 3 分钟讲一下这个项目的核心业务流程和你负责的边界。
2. 为什么把统一鉴权放在 Gateway 做，而不是每个微服务各自做一次？
3. 你在网关里如何区分白名单接口和受保护接口？白名单误配有什么后果？
4. JWT 在网关校验通过后，为什么还要透传 `X-User-Id / X-Username / X-Roles`？
5. 管理端接口为什么要做二次 `ADMIN` 校验（Gateway 一次 + Admin 服务一次）？
6. 如果攻击者绕过 Gateway 直连下游服务伪造 `X-User-Id`，你如何防护？
7. 你们当前 JWT 方案是否支持“主动下线/踢人”？如果不支持，你怎么改？
8. 登录失败锁定是怎么做的？为什么选择 Redis 计数 + 临时锁键？
9. 登录失败阈值（5 次/10 分钟）和锁定时间（5 分钟）是怎么定的？
10. 任务创建时为什么要调用 auth-service 校验 owner 存在，而不是只信前端传参？
11. 任务检索为什么使用 MyBatis-Plus 动态条件而不是手写 SQL 拼接？
12. 关键词检索（title/description/tech_stack）如果数据量变大，like 查询会有什么瓶颈？
13. 你们分页参数做了哪些边界保护？为什么 size 要限制最大值？
14. 任务详情接口里“查看即更新 progress/updatedAt”这个业务设计有什么利弊？
15. 热门任务为什么选 Redis ZSet？与 List/Hash 相比优势是什么？
16. 你们热榜分数是如何计算的？各因子权重（质量/进度/评论/优先级）为什么这么定？
17. 热榜空时触发 `rebuildHot()`，高并发下会不会出现多个请求同时重建？如何避免？
18. 你们是怎么处理缓存穿透、击穿、雪崩的？分别对应哪段实现？
19. `CacheClient` 里互斥锁采用 `setIfAbsent + TTL`，有哪些边界风险？
20. 线程池并行回填热榜详情时，为什么选择 `CallerRunsPolicy`？满载时会有什么表现？
21. 管理看板聚合接口中，如果某个下游 Feign 超时，你希望看板整体失败还是部分可用？为什么？
22. 你当前 Admin 聚合是否做了降级/超时/重试？如果没有，生产上怎么补齐？
23. Nacos 在你们项目中具体承担了哪些职责（注册发现、配置管理）？
24. 配置中心不可用时，服务启动和运行会发生什么？你们如何兜底？
25. 你如何证明这个系统支持“用户侧高频查询 + 管理侧聚合读取”而不把数据库打垮？

## 五、自测建议（回答结构）
1. 每题按“业务目标 -> 当前实现 -> 风险点 -> 优化方案 -> 量化收益”回答。
2. 对你简历里的每个关键词（JWT、Redis ZSet、Feign、Nacos）都准备一个可落地例子。
3. 至少准备 3 个线上故障预案：鉴权异常、缓存故障、下游超时。
