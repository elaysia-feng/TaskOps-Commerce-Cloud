### 1.高频筛选怎么实现（任务分页+动态条件）

  - 在 task-service 用 MyBatis-Plus 的 LambdaQueryWrapper 动态拼条件：关键词（标题/描述/技术栈）、状态、优先级区间。
  - 分页参数有保护：page >= 1，size 限制最大 50，避免大页拖垮 DB。
  - 查询统一按 updatedAt 倒序，适合高频“最近活跃任务”筛选。

### 2.降低数据库热点读怎么实现（热门任务）

  - 首页热榜先读 Redis ZSet：task:hot:zset，直接取 Top10（reverseRange）。
  - 缓存没命中时才重建热榜（最近 200 条任务重算分值写回 ZSet），并且有定时任务每 5 分钟刷新。
  - 热榜详情回填时走 CacheClient.queryWithMutex，做了三件事：
  - 空值缓存防穿透（__NULL__）。
  - 互斥锁防击穿（setIfAbsent + TTL）。
  - TTL 随机抖动防雪崩。![局部截取_20260227_230000](E:\腾讯电脑管家截图文件\局部截取_20260227_230000.png)

### 3.支付功能怎么完成的

  - 订单侧先创建 PENDING_PAY 订单并扣库存。
  - 支付侧提供回调接口（当前是 mock 回调），将支付单置为 SUCCESS。
  - 支付成功后通过 OpenFeign 调订单服务内部接口 mark-paid，把订单状态改为 PAID。
  - 支付侧还有幂等保护：createPaymentIfAbsent 里通过消费日志表 + orderNo 唯一判断避免重复创建支付单。
  - 管理端看板再聚合订单/支付汇总数据展示。

### 4. nacos不可用怎么办

    1. 配置兜底

  - 关键配置本地默认值 + 环境变量兜底，不把生死配置只放 Nacos。

    2. 高可用部署

  - Nacos 集群 + MySQL 持久化 + 负载均衡，不用单机。

    3. 客户端容错

  - 服务调用加超时、重试、熔断、降级，避免把“Nacos抖动”放大成全链路雪崩。

    4. 运维预案

  - 监控 Nacos 可用性、注册实例数、配置推送失败率；异常时切只读/冻结变更，优先保运行稳定。

### 6. nacos怎么实现注册发现和配置管理的

  1. 注册

  - 服务启动后，把自己（服务名、IP、端口）报到 Nacos。

  2. 发现

  - 其他服务调用时只写服务名，不写死 IP，Nacos 返回可用实例地址。

  3. 配置管理

  - 配置放到 Nacos 统一管理，服务启动时拉取；改配置更集中。

### 7. token是怎么传递下去的

  1. 客户端先请求网关。
  2. 网关 GlobalFilter 做登录校验（比如验 JWT），校验通过后把用户信息写到请求头。
  3. 下游微服务收到请求后，用拦截器读取请求头里的用户信息，放到 ThreadLocal，业务代码就能直接取当前用户。
  4. 如果该服务再用 OpenFeign 调别的服务，RequestInterceptor 会把当前用户信息继续放到新请求头里。
  5. 这样用户上下文就能在多个服务间持续透传。

### 8. 做限流时怎么拿到ip地址的

    1. 先从请求头拿真实 IP：X-Forwarded-For（取第一个）。
    2. 没有就拿 X-Real-IP。
    3. 再没有就用 getRemoteAddr() 兜底。

### 9. SprinngMvc

Spring MVC 是 Spring 提供的 Web 框架，用来处理 HTTP 请求。

一次请求的典型流程：

1. 请求先到 **`DispatcherServlet`**（统一入口/前端控制器）
2. 它根据 URL/方法找到对应的 **Controller 方法**（`@GetMapping/@PostMapping`）
3. 自动完成 **参数绑定**（`@RequestParam/@PathVariable/@RequestBody`）和校验
4. 执行 Controller → Service
5. 把返回值通过 **消息转换器**转成响应（常见是 JSON，`@RestController`），写回客户端
