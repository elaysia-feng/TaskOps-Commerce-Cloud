# Redis 三大缓存问题与对应解决方案

## 1) 缓存穿透（Penetration）
- 现象：请求的 key 在缓存和数据库都不存在，导致每次都打到数据库。
- 常见原因：恶意攻击（随机 id）、业务参数越界、热点不存在数据反复查询。
- 解决方式：
  - 布隆过滤器（先判断 key 是否可能存在，不存在直接拦截）
  - 空值缓存（数据库查不到也缓存一个空对象，短 TTL）
  - 参数校验（非法 id 直接拦截）
- 本项目已实现：
  - `task-service` 中 `CacheClient` 使用空值缓存（`__NULL__` + 短 TTL）

## 2) 缓存击穿（Breakdown）
- 现象：某个热点 key 在失效瞬间被大量并发访问，瞬间都回源打数据库。
- 常见原因：热点数据 TTL 到期、并发高峰集中访问单 key。
- 解决方式：
  - 互斥锁（只有一个线程重建缓存，其他线程等待/重试）
  - 逻辑过期 + 后台异步重建
  - 热点 key 永不过期（配合主动更新）
- 本项目已实现：
  - `task-service` 中 `CacheClient` 使用互斥锁重建缓存（`setIfAbsent` + double check）

## 3) 缓存雪崩（Avalanche）
- 现象：大量 key 在同一时间集中过期，导致流量同时回源数据库。
- 常见原因：批量 key 使用相同 TTL、Redis 故障或网络抖动。
- 解决方式：
  - TTL 随机抖动（给过期时间加随机值，打散过期点）
  - 多级缓存（本地缓存 + Redis）
  - 限流/降级/熔断（保护数据库）
  - Redis 高可用（哨兵/集群）
- 本项目已实现：
  - `task-service` 中 `CacheClient` 对缓存 TTL 增加随机抖动（jitter）

---

## 一句话记忆
- 穿透：查不存在 -> 用布隆/空值。
- 击穿：热点 key 失效并发打爆 -> 用锁/逻辑过期。
- 雪崩：大量 key 同时失效 -> 用 TTL 打散 + 限流降级 + 高可用。
