# day9 - Seata 和 MQ 是干嘛的，什么时候加

你说的“两个组件”，这里按当前项目语境指：
- Seata（分布式事务）
- RabbitMQ（消息队列）

## 1. Seata 是干嘛的

核心作用：保证一条业务链路里多个服务/多个数据库操作的事务一致性。

典型场景：
- 下单 -> 扣库存 -> 扣余额，这几个步骤必须“要么都成功，要么都回滚”。

你什么时候需要加 Seata：
- 存在跨服务、跨库写操作，且要求强一致。
- 业务不能接受“部分成功、部分失败”。

你什么时候不要先加 Seata：
- 只是单服务单库事务。
- 业务可以接受最终一致（可补偿）。
- 你现在只是做 MQ 入门演练。

## 2. RabbitMQ 是干嘛的

核心作用：服务之间解耦、异步处理、削峰填谷、事件通知。

典型场景：
- 下单后发一条消息，通知任务服务、通知支付服务，各自异步处理。

你什么时候需要加 MQ：
- 想把同步调用改为异步。
- 需要提高系统吞吐，避免接口链路过长。
- 需要事件驱动（一个动作触发多个后续动作）。

你什么时候不要先加 MQ：
- 当前链路简单，直接同步调用更清晰。
- 还没有明确“谁发、谁收、消息体是什么”。

## 3. 一眼区分（实战判断）

- Seata：解决“跨服务写操作的一致性问题”。
- MQ：解决“服务间通信方式与吞吐问题”。

一句话：
- 你担心“数据回滚一致” -> 想 Seata。
- 你想“异步解耦通知” -> 想 MQ。

## 4. 在你这个项目里的建议（当前阶段）

day9 先做最小可跑：
1. `order-service` 发 MQ 消息
2. `task-service` 收 MQ 消息
3. `pay-service` 先不接 MQ（除非你确实要它消费）
4. Seata 保持你原有链路，不要因为学 MQ 把所有服务都硬加 Seata/MQ

## 5. 常见误区

1. “全服务都加 Seata + MQ 最保险”
- 结果：配置复杂、排错困难、启动依赖变多。

2. “Seata 能替代 MQ”
- 不行。Seata是事务一致性，MQ是消息通信机制，职责不同。

3. “MQ 加了就自动可靠”
- 还需要设计重试、幂等、死信等机制（后续再学）。

---

当前最实用结论：
- 先把 MQ 只加到真正发/收的服务。
- Seata 只保留在确实要跨库一致性的业务链路。
